# Инструкция по тестированию изменений (2024-12-17)

## 🚀 Подготовка

### 1. Запустить Backend
```bash
cd c:\final\backend
python -m uvicorn app.main:app --reload --host 0.0.0.0 --port 18000
```

### 2. Запустить Frontend
```bash
cd c:\final\frontend
npm run dev
```

### 3. Открыть браузер
- URL: `http://localhost:5173`
- DevTools: F12 → Console (для отслеживания ошибок)

## ✅ Тест 1: SSOT Service Code Mappings

### Цель: Проверить что названия услуг отображаются корректно

**Шаги:**
1. Войти как **Регистратор** (registrar/registrar)
2. Открыть **Панель Регистратора**
3. В таблице записей найти любую запись с услугами
4. Проверить что в колонке "Услуги" отображаются **читаемые названия** (не коды):
   - ✅ Правильно: "Консультация кардиолога", "ЭКГ", "ОАК"
   - ❌ Неправильно: "K01", "consultation.cardiology", "1"

**Ожидаемый результат:** Все услуги отображаются человеко-читаемыми названиями.

---

## ✅ Тест 2: Унификация источников услуг (AppointmentWizardV2)

### Цель: Проверить редактирование существующей записи

**Шаги:**
1. В таблице записей найти **QR-запись** (метка "QR" или source='online')
2. Нажать на строку → открыть мастер редактирования
3. **Проверить что в корзине (шаг 2) отображаются исходные услуги:**
   - Услуги должны быть предзаполнены
   - Количество должно соответствовать оригиналу
4. Добавить новую услугу в корзину
5. Нажать "Сохранить"

**Ожидаемый результат:**
- Корзина загружается с правильными услугами
- Редактирование проходит без ошибок
- Новая услуга добавляется к существующим

**Проверить в Console:**
- Нет ошибок `Cannot read properties of undefined`
- Нет ошибок `'api' is not defined`

## ✅ Тест 3: Агрегация пациентов

### Цель: Проверить что все записи пациента отображаются

**Шаги:**
1. Создать пациента с **несколькими услугами одной специальности**:
   - Например: 2 консультации кардиолога
2. Открыть "Все отделения" в панели регистратора
3. Найти этого пациента

**Ожидаемый результат:**
- В строке пациента видны **ВСЕ** его записи
- Номера в очереди отображаются для **каждой** услуги
- Ничего не "потерялось" при агрегации

---

## ✅ Тест 4: Расчёт длины онлайн-очереди

### Цель: Проверить корректность отображения количества людей в очереди

**Подготовка:**
1. Создать несколько пациентов через QR-регистрацию

**Шаги:**
1. Сканировать QR-код клиники (или открыть страницу QR-регистрации)
2. Выбрать врача/специальность
3. Проверить отображаемое "Количество людей в очереди"

**Ожидаемый результат:**
- Число соответствует количеству записей OnlineQueueEntry со статусом `waiting` или `called`
- Число НЕ включает Visit/Appointment из других источников

## ✅ Тест 5: Batch API (опционально)

### Цель: Проверить работу нового Batch API

**Через DevTools Console:**
```javascript
// Импорт не нужен - используем fetch напрямую
const token = localStorage.getItem('access_token');

// GET: Получить все записи пациента за дату
fetch('/api/v1/registrar/batch/patients/1/entries/2024-12-17', {
  headers: { 'Authorization': `Bearer ${token}` }
}).then(r => r.json()).then(console.log);
```

**Ожидаемый результат:**
- API возвращает `{ patient_id, date, online_queue_entries, visits, aggregated }`
- Нет ошибок 500/404

## ✅ Тест 6: Отмена записи

### Цель: Проверить что отмена работает корректно
