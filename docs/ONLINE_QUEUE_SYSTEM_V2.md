## 1. Обзор системы

### Что такое система очередей?

Система очередей — это способ организовать поток пациентов в клинике. Она позволяет:
- Пациентам записаться заранее через QR-код или Telegram
- Регистратору управлять потоком пациентов
- Врачам видеть, кто следующий
- Кассе знать, что оплачено

### Как пациент попадает в очередь?

Существует **5 способов** записи в очередь:

| Способ | Описание | Когда используется |
|--------|----------|-------------------|
| **QR-код** (`online`) | Пациент сканирует QR, выбирает врачей | Утром до 09:00 |
| **Регистратура** (`desk`) | Регистратор добавляет пациента вручную | В любое время |
| **Telegram** (`telegram`) | Пациент записывается через бота | В любое время |
| **Мобильное приложение** (`mobile_app`) | Через PWA-приложение | В любое время |
| **Утреннее распределение** (`morning_assignment`) | Автоматически для подтверждённых визитов | В 06:00 |

## 2. Временные правила

### Когда можно записаться через QR?

| Время | Что происходит |
|-------|----------------|
| **06:00** | Система автоматически создаёт очереди для подтверждённых визитов |
| **07:00–09:00** | Пациенты могут сканировать QR и записываться на СЕГОДНЯ |
| **После 09:00** | QR-запись на сегодня блокируется |
| **После входа регистратора** | QR-запись блокируется (регистратор начал работу) |

### Что такое "блокировка QR"?

Когда регистратор входит в систему утром, QR-запись автоматически блокируется. Это нужно, чтобы:
- Не было хаоса с записями
- Регистратор контролировал поток
- Пациенты, пришедшие позже, обращались в регистратуру

> **Для разработки:** Есть режим Dev Mode (кнопка в Admin Panel или переменная окружения `DISABLE_QUEUE_TIME_RESTRICTIONS=true`), который отключает временные ограничения для тестирования.

## 3. Номера в очереди

### Как присваиваются номера?

- Каждый врач имеет **свой отдельный счётчик** номеров
- Все начинают с номера **1** каждый день
- Кардиолог: 1, 2, 3, 4...
- Стоматолог: 1, 2, 3, 4...

### Что такое `queue_time`?

`queue_time` — это **момент, когда конкретная запись появилась в очереди**.

**Правило:**
- Первая запись → время QR/регистрации
- Новая услуга при редактировании → время редактирования

**Почему это важно:**
- Определяет справедливый порядок в очереди
- Чем раньше `queue_time` — тем раньше пациент будет вызван
- При редактировании старые услуги сохраняют своё время

---

## 4. Порядок в очереди (приоритеты)

### Кто первый?

Порядок определяется **только по `queue_time`**. Источник записи не влияет.

**Пример на 09:30:**
1. Иванов — `morning_assignment` (06:00) → **первый**
2. Петров — `online` (07:15) → **второй**
3. Сидоров — `online` (07:20) → **третий**
4. Козлов — `desk` (09:25) → **четвёртый**

**Почему записанные заранее первые:**
Утреннее распределение происходит в 06:00, поэтому эти пациенты автоматически получают самое раннее `queue_time`.

## 5. Статусы записи в очереди

### Жизненный цикл записи

```
waiting → called → in_service → diagnostics → in_service → served
                       ↘ no_show
                       ↘ incomplete (врач подтверждает)
                       ↘ cancelled (на любом этапе)
```

### Описание статусов

| Статус | Описание |
|--------|----------|
| `waiting` | Пациент ожидает вызова |
| `called` | Вызван (номер на табло, идёт к кабинету) |
| `in_service` | Зашёл к врачу, услуга началась |
| `diagnostics` | Временно на обследовании (ЭКГ, анализы), вернётся |
| `served` | Обслужен и вышел |
| `incomplete` | Ушёл, не вернулся (врач подтверждает с причиной) |
| `no_show` | Не пришёл после вызова |
| `cancelled` | Отменён |

### Статус `diagnostics` — важные правила

**Сценарий:** Врач осмотрел пациента и отправил на ЭКГ. Пациент ушёл на 15–20 минут.

**Что происходит:**
- Статус → `diagnostics`
- Очередь не меняется
- Номер сохраняется
- Визит не закрывается
- EMR остаётся открытым

**Когда пациент вернулся:**
- Врач нажимает "Вернулся в кабинет"
- Статус → `in_service`

### Статус `incomplete` — пациент не вернулся

**Если пациент ушёл домой и не вернулся:**
1. Система НЕ закрывает визит автоматически
2. В конце смены врач получает уведомление
3. Врач выбирает "Завершить как incomplete" с причиной:
   - "Пациент ушёл домой, не вернулся"
   - "Отказался продолжать осмотр"
   - "Не дождался результатов"

**На следующий день:**
- Создаётся НОВЫЙ визит
- Создаётся НОВАЯ запись в очереди
- Старый визит остаётся `incomplete` навсегда
- Старый EMR — только для чтения

## 6. Дубликаты записей

### Что считается дубликатом?

**Для онлайн-записи (QR):**
- Один пациент + один врач + одна услуга + один день = **дубликат (блокируется)**

**Для регистратора:**
- Без ограничений (действует осознанно, несёт ответственность)

### Можно ли записаться к одному врачу на разные услуги?

**Да!** Если это разные услуги — это НЕ дубликат.

Пример: Пациент записался к кардиологу на консультацию. Потом хочет добавить ЭхоКГ. Это разрешено — создаётся вторая запись в очереди.

## 7. Множественная запись через QR

### Как это работает?

1. Пациент сканирует **общий QR клиники**
2. Вводит ФИО и телефон
3. Выбирает нескольких врачей: Кардиолог ✅, Лаборатория ✅
4. Нажимает "Зарегистрироваться"

**Результат:**
- Создаются **2 отдельные записи** в очередях
- Кардиолог: номер 3, `queue_time = 07:30`
- Лаборатория: номер 5, `queue_time = 07:30`
- Одинаковое `queue_time` — справедливо, пациент записался одновременно

## 8. Редактирование визитов — Полная документация

### 8.1 Матрица прав редактирования

| Этап | Регистратор | Врач | Пациент |
|------|-------------|------|---------|
| **До оплаты** | ✅ Полностью: услуги, даты, врачей, комментарии | ❌ Нет доступа | ✅ Свои данные (контакт) |
| **После оплаты, до входа** | ✅ Услуги + корректировка кассы | ❌ Нет доступа | ❌ Нет доступа |
| **Во время приёма (`in_service`)** | ✅ Корректировка визита | ✅ Услуги, EMR, статусы | ❌ Нет доступа |
| **На обследовании (`diagnostics`)** | ✅ Корректировка назначений | ✅ Добавить исследования | ❌ Нет доступа |
| **После завершения (`served`)** | ❌ Запрещено | ❌ Запрещено | ❌ Нет доступа |

**Ключевые правила:**
- **Регистратор** — главный оператор до `served`
- **Врач** — только во время своего приёма
- **Пациент** — минимальные права, только свои данные до оплаты
- После `served` — история неизменяема (для аудита)

### 8.2 Изменение врача/даты после оплаты

**Правило:** Только в особых случаях:
- Болезнь врача
- Форс-мажор
- Системная ошибка

**Обязательные действия:**
1. Уведомление пациенту
2. Корректировка биллинга (доплата/возврат)
3. Логирование в аудите

### 8.3 Действия врача во время приёма

| Действие | Разрешено? | Комментарий |
|----------|------------|-------------|
| Добавить свою услугу | ✅ Да | Врач может добавить услуги, которые сам оказывает |
| Добавить услугу другого врача | ❌ Нет | Через регистратуру |
| Удалить/отменить услугу | ⚠️ Только свои | Изменения логируются |
| Изменить диагноз (EMR) | ✅ Да | Основная задача врача |
| Направить на обследование | ✅ Да | Создаётся новая запись в очереди |
| Отметить услугу как выполненную | ✅ Да | После оказания услуги |

### 8.4 Сценарий: Направление на обследование

**Пример:** Врач-кардиолог говорит: "Вам нужно ЭКГ, потом вернитесь".

**Процесс:**
1. Врач нажимает "Направить на ЭКГ"
2. Статус визита → `diagnostics`
3. Пациент идёт в **регистратуру**
4. **Оплачивает** услугу ЭКГ
5. Получает **талон** с номером очереди на ЭКГ
6. После ЭКГ возвращается к врачу

**Почему через регистратуру?**
- Контроль оплаты
- Печать талона
- Единая точка учёта

### 8.5 Добавление услуги регистратором

**При добавлении новой услуги:**
- Старые записи — **не меняются** (номер, `queue_time` сохраняются)
- Новая услуга → **новый номер в конец очереди** соответствующего врача
- Новая услуга → **`queue_time` = время редактирования**

**Пример:**
- 07:30: Пациент записался на кардиолога (номер 3, `queue_time = 07:30`)
- 14:10: Регистратор добавляет лабораторию
- Результат: Лаборатория (номер 12, `queue_time = 14:10`)

### 8.6 Удаление услуги

**Результат:**
- `OnlineQueueEntry` → статус `cancelled`
- Визит остаётся в системе (история)
- **Физическое удаление НЕ происходит**

### 8.7 Сценарии возврата оплаты

| Ситуация | Действие |
|----------|----------|
| **Отмена до прихода** | Оплата в клинике → нечего возвращать. Онлайн → частичный возврат (- банковская комиссия) |
| **Врач отменяет услугу** | Возврат полной стоимости услуги ИЛИ минус комиссия за консультацию |
| **Замена на более дорогую** | Доплата разницы (старая не возвращается) |
| **Замена на более дешёвую** | Возврат разницы |

## 8.8 ✅ (РЕШЕНО) Проблема: Услуги QR-записей не загружаются в мастере

### Описание проблемы (Историческое)

При редактировании QR-записи в `AppointmentWizardV2`:
1. Услуги не отображаются в корзине (шаг 2)
2. При сохранении создаётся **дубликат** вместо обновления
3. Происходит даже при простом редактировании данных пациента

### Реализованное решение

**1️⃣ Backend: Нормализация API**
API возвращает полные данные о сервисах (`service_details`), включая `service_id`.

**2️⃣ Frontend (RegistrarPanel.jsx): Передача полных данных**
`queue_numbers` обогащается данными из API, передавая `service_id` и `service_name`.

**3️⃣ Frontend (AppointmentWizardV2.jsx): Умный fallback и SSOT**
Мастер использует `updateOnlineQueueEntry` для обновления существующей записи вместо создания новой.

**4️⃣ Дедупликация**
Внедрена строгая проверка наличия записи перед созданием новой, и визуальная группировка в таблице.

### Статус: ✅ Реализовано (v2.2)

## 8.9 ⚠️ ЗАКОН: Отображение услуг в Registrar Panel

### 🚨 КРИТИЧЕСКОЕ ПРАВИЛО

**Единственный источник истины для услуг QR-записей — массив `queue_numbers`!**

```

✅ ПРАВИЛЬНО: Извлекать услуги из queue_numbers и конвертировать в коды
```

### Архитектура

```

│                    RegistrarPanel.jsx                       │
├─────────────────────────────────────────────────────────────┤
│  filterServicesByDepartment(appointment, departmentKey)     │

│  │   └── Собирает ВСЕ услуги из queue_numbers → [K01, S01]  │
│  ├── departmentKey = "cardio"                               │
│  │   └── Находит queue_number по specialty → [K01]          │
│  └── departmentKey = "dental"                               │
│      └── Находит queue_number по specialty → [S01]          │
├─────────────────────────────────────────────────────────────┤
│  filteredAppointments (useMemo)                             │
│  ├── Специализированные вкладки:                            │
│  │   services = filterServicesByDepartment(appt, activeTab) │
│  └── "Все отделения":                                       │
│      aggregatedPatients.map(p => ({                         │
│        ...p,                                                │
│        services: filterServicesByDepartment(p, null) ← ⚠️   │
│      }))                                                    │
└─────────────────────────────────────────────────────────────┘
```

### Функция toServiceCode

Конвертирует любое значение (specialty, service_name, queue_tag) в код услуги:

```javascript
// Примеры конвертации:
"cardiology"    → "K01"
"stomatology"   → "S01"
"dermatology"   → "D01"
"laboratory"    → "L01"
"кардиология"   → "K01"  // Русские названия тоже
"K01"           → "K01"  // Уже код — без изменений
```

### ⚠️ Что НЕ делать

1. **НЕ возвращать `appointment.services` напрямую для QR-записей** — они могут быть неполными или содержать сырые названия.
2. **НЕ забывать вызывать `filterServicesByDepartment`** для агрегированных пациентов в "Все отделения".
3. **НЕ показывать сырые specialty** ("cardiology", "stomatology") — всегда конвертировать в коды.

### Проверка корректности

При изменении логики убедитесь:
- [ ] Пациент с 2+ услугами (QR) в "Все отделения" показывает **ВСЕ** коды (например, `K01, S01`)
- [ ] Специализированные вкладки показывают только **релевантный** код
- [ ] Сырые названия **нигде** не появляются

## 9. Отмена записей

### Кто может отменить?

| Кто | Что может отменить |
|-----|-------------------|
| **Регистратор** | Любые записи |
| **Врач** | Свои записи |
| **Пациент** | Свои записи (через QR/приложение/Telegram) |
| **Система** | Автоматически по таймауту → `no_show` |

### Отмена всего визита

**Результат:**
- Все `OnlineQueueEntry` → `cancelled`
- Визит → `cancelled`
- История сохраняется для аудита

### Отмена одной услуги из нескольких

**Результат:**
- Только эта `OnlineQueueEntry` → `cancelled`
- Остальные услуги остаются активными
- Визит продолжается

**⚠️ ВАЖНО для таблицы "Все отделения":**
В агрегированном виде (вкладка "Все отделения") кнопка "Отмена" **удаляет ВСЕ услуги** пациента за этот день (так как они сгруппированы в одну строку).
Для удаления *конкретной* услуги нужно зайти в режим редактирования (карандаш) → `AppointmentWizardV2`.

### Техническая реализация: `aggregated_ids`

При загрузке данных (`loadAppointments`) записи дедуплицируются по ключу `dedupKey` (patient + date).
Для сохранения всех ID объединяемых записей используется массив `aggregated_ids`:

```javascript
// При создании записи:
aggregated_ids: [entryId]  // например: [407]

// При добавлении дополнительной очереди:
existingAppointment.aggregated_ids.push(entryId);  // [407, 408, 409]
```

**Поток данных:**
1. `loadAppointments` → первая дедупликация (по `dedupKey`) → собирает `aggregated_ids`
2. `mergeAppointments` → вторая дедупликация → объединяет `aggregated_ids`
3. `aggregatePatientsForAllDepartments` → агрегация для вкладки → собирает `aggregated_ids`
4. `onCancel` → итерирует по `aggregated_ids` и отменяет каждую запись

## 10. Оплата и очередь

### Когда платить?

**Правило:** Оплата происходит **ДО входа к врачу** (после регистрации, перед вызовом).

### Сценарии оплаты

| Ситуация | Что происходит |
|----------|----------------|
| **Врач отменяет услугу** | Автоматически создаётся возврат, касса подтверждает |
| **Врач добавляет услугу (у себя)** | Оплата ПОСЛЕ завершения визита |
| **Врач добавляет услугу (в другой очереди)** | Пациент идёт в кассу доплатить ДО |
| **Замена услуги (дороже)** | Старая не возвращается, доплачивается разница |
| **Замена услуги (дешевле)** | Создаётся возврат разницы |

### Пример: Замена консультации на расширенную

- Старая услуга: Консультация (50,000) → `cancelled` (оплата НЕ возвращается)
- Новая услуга: Расширенная консультация (80,000)
- Касса видит: "Доплата 30,000 сум"

## 12. Утреннее распределение (Morning Assignment)

### Как это работает?

**Каждый день в 06:00:**
1. Система берёт все подтверждённые визиты на сегодня
2. Автоматически создаёт записи в очередях
3. Распределяет по врачам
4. Присваивает номера (начиная с 1)

**Результат:**
- Записанные заранее пациенты получают `queue_time = 06:00`
- Они автоматически первые в очереди
- Пациенты из QR (07:00–09:00) встают после них

## 5. Логика дубликатов

### В `check_uniqueness()`:

```python
def check_uniqueness(cls, db, daily_queue, phone, telegram_id, source="online"):
    # Пропускаем проверку для доверенных источников
    if source in ["desk", "morning_assignment"]:
        return None, ""
    
    # Проверяем по телефону
    if phone:
        existing = db.query(OnlineQueueEntry).filter(
            OnlineQueueEntry.queue_id == daily_queue.id,
            OnlineQueueEntry.phone == phone,
            OnlineQueueEntry.status.in_(["waiting", "called"])
        ).first()
        if existing:
            return existing, f"телефону {phone}"
    
    return None, ""
```

## 20. Углубленные сценарии (Edge Cases)

### 20.1 Опоздание пациента (`no_show`)

**Таймаут:**
- Определяется врачом **вручную**.
- Если пациента нет, врач вызывает следующего (текущий помечается как `no_show`).

**Восстановление опоздавшего:**
- Если пациент пришёл после того, как его пропустили:
- **Действие:** Врач или регистратор восстанавливает запись.
- **Приоритет:** Пациент становится **следующим** (высокий приоритет), чтобы зайти сразу после текущего пациента.

### 20.2 Расчет времени ожидания

**Метод:**
- Используется подход **"По количеству людей"**.
- Уведомления: "Перед вами в очереди **3 человека**".
- Расчет по минутам не используется из-за непредсказуемости живой очереди.

### 20.3 Окончание смены / Форс-мажор

**Ситуация:** Смена врача закончилась или ему срочно нужно уйти, но в очереди остались люди.

**Выбор пациента:**
1. **Перенос на завтра:** Пациент переносится на завтрашнее утро и становится **первым** в очереди (после утреннего распределения).
2. **Отмена с возвратом:** Полный возврат средств.

### 20.4 Возврат при онлайн-отмене

**Ситуация:** Пациент оплатил онлайн и отменяет через Telegram/Приложение.

**Выбор пациента:**
1. **Ручной возврат:** Создается заявка, менеджер связывается и переводит деньги на карту.
2. **Депозит:** Деньги остаются на балансе пациента в клинике для оплаты будущих визитов.

### 20.5 Тип очереди

- **Строго "Живая очередь"**.
- Никаких временных слотов (например, "на 10:30").
- Все пациенты идут в порядке `queue_time` и приоритетов восстановления.

## 21. Редактирование и предотвращение дублей QR-записей

Введена строгая логика "Единого источника истины" (SSOT) для редактирования QR-записей, чтобы предотвратить создание дубликатов.

### Ключевые принципы

1.  **Запрет создания новых записей при редактировании**
    *   При редактировании существующей QR-записи (`record_type='online_queue'`, `source='online'`) **ЗАПРЕЩЕНО** использовать endpoint создания (`POST /registrar/cart`).
    *   Вместо этого данные должны обновляться в исходной записи.

2.  **Использование Full Update Endpoint**
    *   Для сохранения изменений используется специальный endpoint:
        `PUT /api/v1/qr-queue/online-entry/{entry_id}/full-update`
    *   Этот метод атомарно обновляет пациента, услуги, тип визита и скидки, сохраняя `queue_time` и позицию в очереди для неизменных услуг.

3.  **Логика Frontend (AppointmentWizardV2)**
    *   Мастер редактирования детектирует QR-запись.
    *   Вызывает функцию `updateOnlineQueueEntry`.
    *   При успехе немедленно завершает работу, предотвращая вызов старой логики создания.

4.  **Дедупликация в UI (RegistrarPanel)**
    *   Записи группируются визуально по ключу: `online_{pid|phone|fio}_{date}`.
    *   Это скрывает технические особенности реализации (когда одна запись может быть разбита на несколько `OnlineQueueEntry` в случае сложной логики, хотя сейчас мы стремимся к одной записи).

### Техническая реализация (SSOT)

*   **`service_id`**: Является каноническим идентификатором услуги.
*   **`updateOnlineQueueEntry`**: Frontend-функция в `api/queue.js`, обертка над `PUT`.
