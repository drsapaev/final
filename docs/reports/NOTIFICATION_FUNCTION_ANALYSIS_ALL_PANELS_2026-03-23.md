## 2. Ключевые слои системы уведомлений

### 2.1 Глобальный UI-слой уведомлений

Во всём приложении глобально подключены:

- `ToastProvider` — собственная внутренняя система toast-уведомлений.
- `NotificationWebSocketProvider` — WebSocket-подключение для уведомлений в реальном времени.
- `NotificationPrompt` — запрос разрешения браузерных уведомлений.

Это означает, что техническая основа для единой системы уведомлений уже присутствует на уровне всего приложения.

### 2.2 Browser / push / chat notifications

Текущая browser/push-логика ориентирована в первую очередь на чат:

- `NotificationPrompt` предлагает включить уведомления именно о новых сообщениях.
- `pushNotifications` показывает системные браузерные уведомления для входящих сообщений и счётчика непрочитанных.

Следствие: клинические панели подключены к инфраструктуре только косвенно; основной продуктовый сценарий push-уведомлений сейчас — чат, а не рабочие события панелей.

### 2.3 WebSocket-уведомления

`NotificationWebSocketProvider` умеет:

- обрабатывать сообщения типа `notification`;
- показывать их через `addToast(...)`;
- при скрытой вкладке дублировать уведомление через `new Notification(...)`;
- отдельно реагировать на `queue_update`.

Но сама интеграция панелей с этим каналом слабая: наличие глобального провайдера не означает, что панели реально публикуют или читают предметные события.

### 2.4 API-слой уведомлений

Во frontend есть общий `notificationsService` для:

- списка уведомлений;
- получения по ID;
- `mark read`;
- `mark all read`;
- отправки уведомления;
- получения типов.

При этом исторический parity-отчёт уже фиксировал несоответствия части frontend notification API backend-контрактам. Практически это выглядит как незавершённая или частично устаревшая унификация.

## 3. Панели: фактическое состояние

Ниже — только то, что подтверждается кодом.

### 3.1 Admin Panel

**Степень зрелости: высокая, но неоднородная.**

Что есть:

- На dashboard загружаются системные уведомления/активность через `/notifications/history/stats?days=7`.
- Есть отдельный раздел `UnifiedNotifications`.
- Внутри `UnifiedNotifications` доступны:
  - `FCMManager` — управление FCM, тестовые push, ручная отправка;
  - `RegistrarNotificationManager` — уведомления регистраторам, статистика, тестовая отправка.

Проблемы:

- Внутри самой админ-панели по-прежнему много `alert(...)` для действий пользователя.
- То есть Admin одновременно использует:
  - dashboard-данные уведомлений,
  - полноценные notification managers,
  - старые browser `alert(...)`.

**Вывод:** админка — единственная панель, где уведомления оформлены как самостоятельная подсистема, но UX всё ещё не стандартизирован.

### 3.2 Registrar Panel

**Степень зрелости: высокая на уровне локального UX, низкая на уровне общей архитектуры уведомлений.**

Что есть:

- Панель активно использует `react-toastify`:
  - вызов пациента;
  - оплата;
  - обновление статуса;
  - перенос визита;
  - экспорт;
  - массовые действия.

Что отсутствует:

- Нет собственного inbox/центра уведомлений.
- Не найдено интеграции с `notificationsService`.
- Не найдено явной привязки к `NotificationWebSocketContext` внутри панели.

**Вывод:** у регистратора много операционных toast-ответов, но это не единая notification-функция, а набор локальных подтверждений действий.

### 3.3 Doctor Panel

**Степень зрелости: низкая.**

Что есть:

- Есть кнопка с иконкой `Bell` для сценария «Вернуть с диагностики (Push)».
- Основная очередь управляется через `useDoctorQueue`.

Что не подтверждено кодом:

- Нет `react-toastify` в файле панели.
- Нет `alert(...)` для штатных уведомлений.
- В `useDoctorQueue` есть только API-вызовы и перезагрузка очереди, без user-facing notification слоя.

**Вывод:** Doctor Panel визуально намекает на push-сценарий, но фактическая пользовательская система уведомлений в коде панели почти отсутствует.

### 3.4 Cardiologist Panel

**Степень зрелости: низкая.**

Что есть:

- Обнаружен только `alert(...)`-placeholder для оплаты пациента.

Что отсутствует:

- Нет toast-системы в файле панели.
- Нет notification center.
- Нет интеграции с общим frontend notifications API.

**Вывод:** кардиологическая панель практически не реализует собственную функцию уведомлений.

### 3.5 Dermatologist Panel

**Степень зрелости: средняя.**

Что есть:

- Активное использование `react-toastify`:
  - загрузка пациента;
  - сохранение рецепта;
  - перенос AI-предложений;
  - завершение приёма;
  - вызов следующего пациента;
  - сохранение осмотров и процедур.

Что отсутствует:

- Нет отдельного notification center/inbox.
- Нет интеграции с глобальным notificationsService.
- Часть UX по-прежнему на `alert(...)` placeholders.

**Вывод:** у дерматолога хорошие локальные toast-уведомления, но нет общей архитектурной интеграции с единой системой уведомлений.

### 3.6 Dentist Panel

**Степень зрелости: низкая-средняя.**

Что есть:

- Используются отдельные `toast.info(...)` / `toast.error(...)` для guard-сценариев, связанных с `visit_id`.

Что отсутствует:

- Нет полноценного notification workflow.
- Нет inbox/истории уведомлений.
- Есть `alert(...)` placeholder для оплаты.

**Вывод:** стоматологическая панель использует уведомления точечно, в основном как валидационные подсказки.

### 3.7 Cashier Panel

**Степень зрелости: низкая.**

Что есть:

- Пользовательские уведомления почти полностью реализованы через browser `alert(...)`:
  - успешная оплата;
  - ошибка обработки;
  - отмена платежа;
  - возврат;
  - экспорт;
  - получение чека;
  - загрузка статистики.

Что отсутствует:

- Нет `react-toastify`/внутреннего ToastProvider-паттерна внутри панели.
- Нет notification center.
- Нет интеграции с общим notification API.

**Вывод:** у кассира уведомления есть, но реализация самая устаревшая и наименее консистентная.

### 3.8 Lab Panel

**Степень зрелости: низкая в shell-панели.**

Что есть:

- Shell `LabPanel.jsx` сам по себе не содержит видимой логики toast/notification.
- Панель больше выступает контейнером для `LabQueueWorkbench`, `LabReportWorkbench`, `LabTemplateWorkbench`.

Что не удалось подтвердить в рамках именно panel-shell анализа:

- Что лабораторные уведомления централизованы в workbench-компонентах.

**Вывод:** на уровне основной панели лаборатории notification-функция почти не проявляется.

### 3.9 Patient Panel

**Степень зрелости: очень низкая / фактически отсутствует.**

Что есть:

- Панель наполнена mock-данными через `setTimeout`.
- В panel-shell нет toast/push/inbox логики.

Что отсутствует:

- уведомления о визитах;
- уведомления о готовности результатов;
- история уведомлений;
- действия `mark as read`.

**Вывод:** Patient Panel пока не выглядит интегрированной в общую notification-функцию.

## 4. Главные архитектурные проблемы

### 4.1 Нет единого UX-стандарта уведомлений

Сейчас одновременно используются:

- собственный `ToastProvider`;
- `react-toastify`;
- browser `alert(...)`;
- browser `Notification` API.

Это создаёт фрагментацию по всем панелям.

### 4.2 Глобальная инфраструктура есть, но панели её почти не потребляют

На уровне providers есть хорошая база:

- WebSocket provider;
- prompt на browser notifications;
- custom toast layer.

Но панельный код в основном живёт отдельно и использует локальные `toast`/`alert`.

### 4.3 Generic notification API не выглядит полностью внедрённым

Есть общий `notificationsService`, но panel-level usage практически не найден. Это типичный признак незавершённой унификации.

### 4.4 Самая развитая notification-функция сосредоточена только в админке

Admin умеет:

- смотреть статистику;
- управлять FCM;
- отправлять уведомления регистраторам;
- тестировать уведомления.

Остальные панели в основном лишь показывают локальные ответы на действия пользователя.

### 4.5 Patient/Lab/Doctor coverage недостаточна

Для клиники особенно важны рабочие уведомления в этих ролях:

- врач — возврат с диагностики, готовность результатов, срочные системные сигналы;
- лаборатория — критические результаты, дедлайны, подтверждение отправки результатов;
- пациент — готовность анализов, перенос визита, подтверждение оплаты, напоминания.

По коду такое покрытие пока не выглядит целостным.

## 6. Рекомендуемый план улучшения

### P1 — унификация UX

1. Выбрать **один** фронтенд-механизм показа уведомлений для всех панелей:
   - либо внутренний `ToastProvider`,
   - либо единообразно `react-toastify`.
2. Удалить `alert(...)` из panel workflows.
3. Завести единый helper уровня `notify.success/error/info/warn`.

### P2 — унификация событий

1. Разделить уведомления по доменам:
   - queue;
   - payment;
   - lab;
   - appointment;
   - admin/system;
   - patient-facing.
2. Привязать panel actions к единым event payloads.
3. Использовать `NotificationWebSocketProvider` не только для чата, но и для клинических событий.

### P3 — inbox / history

1. Добавить общий notification inbox хотя бы для:
   - registrar;
   - doctor;
   - lab;
   - patient.
2. Реально задействовать `notificationsService` там, где backend уже даёт историю/отправку/шаблоны.
3. Проверить и закрыть backend/frontend parity для `GET /notifications`, `mark read`, `mark all read`, `types`.

### P4 — role-specific coverage

1. **Doctor:** вернуть с диагностики, критические результаты, изменение очереди.
2. **Lab:** готовность результата, критическое отклонение, подтверждение отправки врачу/пациенту.
3. **Patient:** напоминание о визите, перенос, оплата, готовность анализов.
4. **Cashier:** заменить все `alert(...)` на структурированные уведомления и статусные banners/toasts.

## 7. Итог

Функция уведомлений **есть в проекте как набор нескольких подсистем**, но **не существует как единая, последовательно внедрённая cross-panel capability**.

Самое сильное место — **Admin Panel**.
Самое операционно насыщенное локальными сообщениями место — **Registrar Panel**.
Самые слабые зоны — **Doctor / Lab / Patient / Cashier** с точки зрения единой архитектуры и согласованного UX.

Если следующий шаг — не просто анализ, а реализация, то оптимальный порядок такой:

1. унифицировать frontend notification layer;
2. убрать `alert(...)` из panel workflows;
3. внедрить inbox + realtime для Doctor / Registrar / Lab / Patient;
4. затем выровнять backend/frontend contract общего notifications API.
