# ADR-0001: Centralize Caching for EMR v2

**Status:** Accepted  
**Date:** 2026-02-06  
**Owner:** EMR Platform

## Context

В старых панелях кэширование реализовывалось локально: локальные `localStorage`-записи, разрозненные TTL, разные ключи и паттерны инвалидации. Это давало временную выгоду, но приводило к расхождениям, трудноотлавливаемым багам и несогласованному поведению между экранами.

Для EMR v2 нужна единая политика кэширования, чтобы:

- единообразно управлять TTL и инвалидацией,
- минимизировать дублирование кода,
- снизить риск утечек данных между визитами,
- обеспечить безопасность медицинских данных при смене пациента/визита.

## Decision

Вводим единый слой кэширования в `frontend/src/core/cache/` и используем его во всех панелях и сервисах:

- единый интерфейс `get/set/invalidateByVisit`,
- единые TTL и теги (visitId/patientId),
- запрет локальных TTL и `localStorage` в компонентах.

## Consequences

Плюсы:

- единая точка контроля кэша и политики,
- предсказуемое поведение по всей системе,
- безопасная инвалидация при смене визита.

Минусы:

- требуется миграция старых/локальных решений,
- новые интеграции должны следовать policy.

## Alternatives Considered

- Локальные кэши в компонентах: быстро, но ведёт к рассинхронизации и дублированию.
- Общие хуки без централизованного сервиса: частично решает, но оставляет расплывчатую ответственность.

## Follow-up

- Мигрировать существующие кэши на единый сервис.
- Добавить проверку в code review: запрещён `localStorage` в компонентах для кэша.
