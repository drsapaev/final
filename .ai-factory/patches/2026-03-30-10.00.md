# EMR v2 prop-types cleanup for migrated primitives

**Date:** 2026-03-30 10:00
**Files:** frontend/src/components/emr-v2/sections/*, frontend/src/components/emr-v2/ai/*, frontend/src/components/emr-v2/DoctorTemplatesPanel.jsx, frontend/src/components/emr-v2/EMRDiffViewer.jsx, frontend/src/components/emr-v2/EMRHelpDialog.jsx, frontend/src/components/emr-v2/EMRHistoryPanel.jsx, frontend/src/components/emr-v2/templates/TreatmentTemplatesPanel.jsx
**Severity:** low

## Problem

ESLint reported 119 `react/prop-types` warnings across the newly migrated EMR v2 primitives and adjacent helper panels. The warnings came from components that were moved out of the legacy `components/emr` tree into the v2 layer without carrying over `propTypes` definitions.

## Root Cause

The EMR migration focused on moving reusable UI pieces and wiring them into the v2 sections, but the prop validation layer was not treated as part of the component contract. As a result, the new v2 primitives rendered correctly, but the lint gate still saw public props as undocumented and flagged every destructured prop.

## Solution

Added `propTypes` to the migrated v2 primitives and their nearby helper panels:
- base wrappers like `EMRSection` and `EMRTextField`
- AI widgets like `AISuggestionCard`, `AISuggestionPanel`, `PhraseSuggestions`, and `SmartAssistButton`
- clinical section primitives like `ComplaintsField`, `ExaminationMatrix`, `VitalsWidget`, `PrescriptionEditor`, `NotesSection`, `RecommendationsSection`, `AnamnesisMorbiSection`, `AnamnesisVitaeSection`, and `ComplaintsSection`
- specialty panels like cardiology, dentistry, and dermatology
- helper panels like `DoctorTemplatesPanel`, `EMRDiffViewer`, `EMRHistoryPanel`, `EMRHelpDialog`, and `TreatmentTemplatesPanel`

Verified the cleanup with a targeted `eslint` run on the full `emr-v2` tree and a production `vite build`.

## Prevention

- When migrating a legacy component into `emr-v2`, carry its public prop contract with it at the same time.
- Run `eslint` on the whole v2 subtree after any component split or move, not just the files that changed directly.
- For small wrapper components, add `propTypes` immediately after the component move so warnings do not accumulate later.
