# Relaxed auth session rate limit and frontend validation dedupe

**Date:** 2026-03-03 11:25
**Files:** backend/app/middleware/security_middleware.py, backend/tests/test_security_middleware.py, frontend/src/stores/auth.js, frontend/src/stores/__tests__/auth.test.js, frontend/src/App.jsx
**Severity:** high

## Problem

Authenticated users started getting `429 Too Many Requests` on `GET /api/v1/auth/me`
with detail `Превышен лимит запросов: 100 за 3600 секунд`. This broke route guards,
caused console noise, and could incorrectly block access even though the session was valid.

## Root Cause

Two issues compounded:

1. Backend rate limiting classified `/api/v1/auth/me` and `/api/v1/authentication/profile`
   as generic `api` traffic, so they shared the low `100/3600` bucket with all other API calls.
2. Frontend `RequireAuth` forced `validateSession(true)` on each protected route mount. In dev,
   nested guards and StrictMode could trigger repeated `/auth/me` checks, consuming the generic bucket.

## Solution

- Added a dedicated `session` rate-limit bucket in `SecurityMiddleware` and classified
  `/auth/me`, `/authentication/profile`, `/authentication/status`, and
  `/authentication/sessions/current` into it.
- Kept logging under `[FIX:RATE_LIMIT]` for easier verification of endpoint classification.
- Removed forced revalidation from `RequireAuth`.
- Added a short in-memory validation cache and single-flight promise in the auth store so
  near-duplicate session checks reuse recent state instead of refetching `/auth/me`.
- Added explicit `[FIX:AUTH]` logging when rate-limited profile checks fall back to cached auth state.

## Prevention

- Session-validation endpoints should not share the same budget as the general API bucket.
- Route guards should avoid unconditional forced revalidation, especially under React StrictMode.
- Keep regression tests for endpoint classification and auth-store validation caching.
