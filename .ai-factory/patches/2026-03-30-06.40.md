# Doctor queue panel used stale backend origin 8000

**Date:** 2026-03-30 06:40
**Files:** frontend/src/components/doctor/DoctorQueuePanel.jsx, frontend/src/components/doctor/__tests__/DoctorQueuePanel.test.jsx, .ai-factory/RULES.md
**Severity:** medium

## Problem

The doctor queue panel was still calling `http://127.0.0.1:8000/api/v1/...` directly. In the current local stack the backend runs on `18000`, so these requests failed at the network layer and surfaced as browser fetch failures.

## Root Cause

The component had a hardcoded legacy backend origin copied from an older local setup. It bypassed the shared runtime URL builder, so it could drift away from the canonical backend port whenever the stack changed.

## Solution

Replaced all direct doctor queue fetch URLs with `buildApiUrl(...)` so the component now derives its backend origin from the shared runtime config. Added structured `[FIX:DOCTOR_QUEUE_PANEL]` logs for the resolved URLs and a regression test that asserts the panel calls the canonical `18000` origin and never reverts to `8000`.

## Prevention

- Never hardcode backend origins or ports in frontend components.
- Use `buildApiUrl` / `buildWsUrl` for all browser network calls.
- Add a regression test whenever a legacy origin or port is removed.
- When smoke fails with a fetch/network error, grep for stale origins before assuming auth or CORS is the cause.
