# React warning and repeated 403 noise in cardiology flow

**Date:** 2026-04-09 15:19
**Files:** `frontend/src/components/ui/macos/Input.jsx`, `frontend/src/pages/CardiologistPanelUnified.jsx`
**Severity:** medium

## Problem

The login form emitted a React dev warning about mixing `border` and `borderColor` on the same input.  
The cardiology panel also kept issuing file-list requests that returned `403 Forbidden`, creating repeated console noise during normal navigation.

## Root Cause

The macOS input component used the `border` shorthand in its base inline styles and later updated `borderColor` during focus and error states. React treats that as a conflicting style update and warns during rerender.

The cardiology panel eagerly tried both patient-scoped and visit-scoped file list endpoints even after the first permission-denied response. Because the endpoint is not accessible for the current role/context, the component kept retrying a request that could never succeed.

## Solution

Replaced the `border` shorthand in the shared input component with explicit `borderWidth`, `borderStyle`, and `borderColor` fields so focus and error updates no longer conflict.

In the cardiology panel, added a permission-denied guard for file fetching:
- stop after the first `403`
- remember that file access is denied for the session
- skip future file-list fetches instead of repeatedly hitting a forbidden endpoint
- log the skip once with a `[FIX:FILES]` prefix for troubleshooting

## Prevention

- Prefer explicit longhand style properties when a component updates a subset of border-related fields on rerender.
- When an endpoint can legitimately return `403`, short-circuit future fetches after the first denial instead of retrying every load.
- Add tests for shared UI primitives that are reused across sensitive flows like auth and EMR.
