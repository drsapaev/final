# Fixed 2FA backup code regeneration flow

**Date:** 2026-03-03 13:17
**Files:** frontend/src/components/security/TwoFactorManager.jsx, frontend/src/components/security/__tests__/TwoFactorManager.test.jsx
**Severity:** medium

## Problem

Clicking "Обновить коды" in the 2FA UI triggered `POST /api/v1/2fa/backup-codes/regenerate`
and returned `400 Bad Request`.

## Root Cause

The frontend treated `/2fa/setup` as if 2FA were already enabled, but the backend only enables
2FA after a separate `POST /2fa/verify-setup`. Because of that mismatch:

- users could reach backup-code actions before 2FA was actually active
- the UI never guided users through the required TOTP verification step
- the backup tab allowed actions that the backend correctly rejected with `400`

## Solution

- Added the missing verify-setup step directly into `TwoFactorManager`
- After `/2fa/setup`, the UI now shows QR/secret and requires a 6-digit TOTP confirmation
- Blocked backup-code regeneration until `status.enabled === true`
- Added explicit loading of existing backup codes and switched the "show codes" action to real data loading
- Added `[FIX:2FA]` logging around setup/verification/loading flows

## Prevention

- Frontend auth/security flows must follow backend state transitions exactly
- UI actions for sensitive operations should be gated by actual server state, not assumed local progress
- Keep regression tests for multi-step security flows, especially setup-vs-enabled transitions
