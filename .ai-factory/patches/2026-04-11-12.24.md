# Registrar mark-paid preserves operational status

**Date:** 2026-04-11 12:24
**Files:** `backend/app/api/v1/endpoints/registrar_wizard.py`
**Severity:** high

## Problem

Registrar mark-paid endpoints wrote `status="paid"` into `Visit.status` and `OnlineQueueEntry.status`. That mixed payment state with operational queue state and could make registrar table actions disappear or show the wrong status column meaning.

## Root Cause

The payment workflow used the same `status` field for two different concepts: payment completion and queue/visit lifecycle. The rest of the system expects payment to be represented through `discount_mode` and `payments`, while queue status should remain operational.

## Solution

Added a narrow normalizer that preserves operational status during payment. If legacy data already has `status="paid"` or a blank status, the endpoint now normalizes it to `waiting`. Mark-paid responses also include `payment_status="paid"` and logging with `[FIX:PAYMENT_STATUS]`.

## Prevention

- Do not write payment state into operational `status` fields.
- Review payment endpoints for separate `payment_status` or `discount_mode` handling.
- Add regression coverage for mark-paid preserving queue actions/status.
