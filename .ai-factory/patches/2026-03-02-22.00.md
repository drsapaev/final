# Patch Note - 2026-03-02 22:00

## Context

Follow-up fix for remaining frontend console errors after auth/Postgres migration:
- chat websocket failing before connection is established
- notification websocket logging noisy error events
- `POST /api/v1/2fa/regenerate-backup-codes` returning `404`
- `POST /api/v1/2fa/setup` returning `500`
- missing `/api/v1/auth/csrf-token` causing repeated `404` warnings in dev

## Changes

### Frontend

- Fixed wrong 2FA backup code endpoint in `frontend/src/components/security/TwoFactorManager.jsx`
  - now uses `/api/v1/2fa/backup-codes/regenerate`
  - added `[FIX:2FA]` log before request

- Reduced false-positive CSRF noise in `frontend/src/api/client.js`
  - cache backend `404` for `/auth/csrf-token`
  - log one `[FIX:CSRF]` info and skip future bootstrap attempts instead of warning every POST

- Hardened websocket lifecycle in `frontend/src/contexts/ChatContext.jsx`
  - read freshest token from `tokenManager` on connect/reconnect
  - avoid closing CONNECTING sockets during StrictMode cleanup in a way that triggers browser console noise
  - retry only while mounted
  - added `[FIX:WS]` connection/reconnect logs

- Reduced notification websocket noise in `frontend/src/contexts/NotificationWebSocketContext.jsx`
  - skip connect while token is missing/expired
  - downgrade handshake failure from error to warn
  - added provider `propTypes`

- Added missing `propTypes` for `ChatProvider` and `NotificationWebSocketProvider`

### Backend

- Fixed 2FA setup persistence in `backend/app/services/two_factor_service.py`
  - `db.flush()` before creating backup/recovery children so `two_factor_auth.id` exists
  - clear previous backup/recovery artifacts on re-setup
  - only create recovery row when recovery email/phone is actually provided
  - populate `backup_codes_generated` and `backup_codes_count`
  - added `[FIX:2FA]` logs

- Relaxed response schema in `backend/app/schemas/two_factor_auth.py`
  - `TwoFactorSetupResponse.recovery_token` and `expires_at` are now optional
  - this matches setup flows where no recovery contact is configured

### Tests

- Extended `backend/tests/unit/test_user_bootstrap_and_two_factor_service.py`
  - verifies 2FA setup works without recovery contact
  - verifies backup codes are persisted and no invalid recovery row is created

## Verification

- `python -m pytest backend/tests/unit/test_user_bootstrap_and_two_factor_service.py -q`
  - result: `3 passed`

- `npx eslint src/components/security/TwoFactorManager.jsx src/api/client.js src/contexts/ChatContext.jsx src/contexts/NotificationWebSocketContext.jsx`

## Expected Outcome

- no repeated CSRF `404` warning spam from `client.js`
- no `404` on backup code regeneration
- `2fa/setup` no longer fails when recovery email/phone is empty
- chat websocket reconnect uses the latest auth token instead of stale store state
- notification websocket no longer logs `[ERROR] Notification WebSocket Error: [Event]` for transient handshake failures
