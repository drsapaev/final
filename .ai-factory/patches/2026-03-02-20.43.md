# Auth guard loop and stale-session 401 storm in protected routes

**Date:** 2026-03-02 20:43
**Files:** frontend/src/App.jsx, frontend/src/stores/auth.js, frontend/src/stores/__tests__/auth.test.js
**Severity:** high

## Problem

Protected routes emitted `Warning: Maximum update depth exceeded` from
`RequireAuth` and then spammed the console with `401 Unauthorized` requests from
pages like registrar and queue tabs.

## Root Cause

Two issues compounded:

1. `RequireAuth` subscribed inside an effect that depended on `st.profile`.
   The auth store immediately invoked subscribers with a freshly parsed profile
   object, so the dependency changed every render and caused a re-subscribe loop.
2. Stale or expired tokens were not invalidated before protected pages mounted.
   Route guards treated the presence of `auth_token` and cached `auth_profile` as
   sufficient, so child pages fired protected API requests and produced a 401
   burst before auth state was cleaned up.

## Solution

Reworked frontend auth flow so protected-route validation happens once per route
transition instead of on every profile object change. Added `validateSession()`
to the auth store with `[FIX:AUTH]` logging, expired-token detection, single
profile-load flow, and atomic clearing of both token and profile on 401/403.
`RequireAuth` now subscribes safely, compares auth snapshots, and validates the
session before rendering nested protected pages.

## Prevention

- Do not use mutable object identities from storage-backed state as `useEffect`
  dependencies for subscription setup.
- Validate protected sessions before mounting heavy route trees that immediately
  start authenticated data fetches.
- Cover auth-store cleanup paths with tests for expired tokens and backend 401
  responses.
