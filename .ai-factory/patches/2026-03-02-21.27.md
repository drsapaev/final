# Patch: Migrate Legacy SQLite Users Into PostgreSQL

Date: 2026-03-02 21:27

## Problem

The project is now Postgres-first, but the active user accounts still lived in
legacy `backend/clinic.db`. After removing the stale IDE SQLite override, the
current Postgres database had only a health-check admin account, so real logins
like `registrar@example.com` could not work against PostgreSQL.

## Root Cause

- Historical users existed only in SQLite.
- Current Postgres `users` table had not been backfilled.
- Login and `/auth/me` behavior diverged depending on whether the backend was
  accidentally started against SQLite or correctly started against Postgres.

## Solution

Added a reusable migration module:

- `backend/app/scripts/migrate_users_to_postgres.py`

The script:
- reads legacy users from `backend/clinic.db`
- preserves legacy `users.id` values for newly inserted rows
- keeps hashed passwords unchanged
- performs safe update-or-insert behavior by `username/email`
- aborts on dangerous ID collisions
- syncs the PostgreSQL `users_id_seq` sequence after explicit ID inserts
- emits `[FIX:USER-MIGRATION]` logs for traceability

Added regression tests:

- `backend/tests/unit/test_migrate_users_to_postgres.py`

## Execution

Ran:

```powershell
cd C:\final\backend
python -m app.scripts.migrate_users_to_postgres
```

## Prevention

- Keep Postgres as the only active development source of truth.
- Avoid hidden launch-time DB overrides.
- Use the migration script for any remaining legacy local backfills instead of
  ad-hoc SQL edits.
