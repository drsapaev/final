# Finance amount field blocked normal UZS values

**Date:** 2026-03-27 09:00
**Files:** frontend/src/components/admin/FinanceModal.jsx, frontend/src/components/admin/__tests__/FinanceModal.test.jsx, .ai-factory/logs/PANEL_QA_IMPLEMENTATION_STATUS.md
**Severity:** low

## Problem

The finance modal accepted valid positive amounts in backend validation, but the browser-side numeric field used `step="1000"`. That caused native `stepMismatch` blocking for ordinary values like `12345`, so the submit button appeared broken even though the form logic itself was fine.

## Root Cause

The amount input was optimized too aggressively for round-number UX and accidentally made the browser validator stricter than the actual finance rules. The frontend validation only required a positive number, but the native input constraint required multiples of 1000, which was inconsistent with real-world finance entry.

## Solution

Relaxed the amount field to integer-friendly validation:
- changed `step` from `1000` to `1`
- aligned `min` with the actual validation (`1`)
- added `inputMode="numeric"` and a short helper hint so users know to enter a whole UZS amount
- added a focused regression test for the amount-field constraints
- added `[FIX:FINANCE-AMOUNT-STEP]` logging around blocked submit and successful submit paths

## Prevention

- Keep native input constraints aligned with actual backend/frontend validation rules
- For currency fields, verify both browser validation and submit flow with a non-round value during QA
- Add a small component test whenever a form constraint is changed
