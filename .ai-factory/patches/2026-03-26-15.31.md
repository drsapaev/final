# Service catalog prefix drift in admin create/update

**Date:** 2026-03-26 15:31
**Files:** `backend/app/services/service_mapping.py`, `backend/app/api/v1/endpoints/services.py`, `frontend/src/components/admin/ServiceCatalog.jsx`, `backend/tests/integration/test_admin_service_code_prefix_consistency.py`
**Severity:** medium

## Problem

Admin users could save a service whose SSOT code prefix did not match the selected routing group. The catalog also displayed legacy `code` values as if they were canonical, so mismatched metadata could stay hidden until downstream routing or editor flows touched the record.

## Root Cause

The service create/update path normalized code format, but it did not validate that the code prefix matched the chosen category/queue. In addition, `category_code` was being normalized through the generic service-code normalizer, which turned single-letter routing tokens into lowercase and made the catalog harder to read consistently.

## Solution

Added server-side prefix guardrails in the service create/update endpoint:
- canonicalize `code` and `service_code` to the same value
- resolve the expected routing group from `queue_tag`, `department_key`, and category specialty
- reject SSOT-style codes when the prefix does not belong to the selected group
- normalize `category_code` to uppercase single-letter tokens

On the frontend, the admin service catalog now:
- warns before submit when the prefix does not match the selected group
- shows the canonical code in the list view
- surfaces legacy mismatches more clearly

The fix includes `[FIX:ADM-06]` logging on the backend so rejected or accepted prefix checks are easy to trace.

## Prevention

- Treat service code prefix alignment as part of service creation, not as downstream cleanup
- Keep routing metadata SSOT-backed and validate it server-side
- Use the canonical code in catalog displays so legacy values do not look authoritative
- Add regression tests for both rejection and allowed multi-prefix groups
