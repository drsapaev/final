# Login error normalization and live-credential guardrail

**Date:** 2026-03-30 06:12
**Files:** `frontend/src/components/auth/LoginFormStyled.jsx`, `frontend/src/components/auth/loginErrorUtils.js`, `frontend/src/components/auth/__tests__/loginErrorUtils.test.js`, `.ai-factory/RULES.md`
**Severity:** medium

## Problem

The login form exposed the raw browser/network failure `Failed to fetch`, which is confusing for users and gives no recovery hint. In the same workflow, agent-driven smoke was also spending time rediscovering live credentials from the database instead of treating live DB values as the source of truth.

## Root Cause

The login flow was passing low-level network exceptions straight through to the UI without normalization. That meant browser fetch failures, proxy issues, or backend downtime all surfaced as the same opaque text. Separately, the AI Factory guidance did not explicitly forbid guessing credentials from stale artifacts, so smoke runs kept re-validating the same mismatch between docs and live DB.

## Solution

Added a small login-error formatter that maps:
- network failures to a clear backend-connection message
- `401` to a credential error
- backend validation details to the returned message

The login form now uses that formatter for both non-OK responses and thrown fetch errors. A regression test covers the network, auth, and backend-detail cases. A project rule was also added so future auth/smoke work resolves working credentials from the live database or live auth response instead of stale token files or docs.

## Prevention

- Never surface raw `Failed to fetch` text to users in auth flows.
- Always normalize login failures into a localized, actionable message.
- Add regression tests for `401`, backend detail text, and network failure cases.
- For smoke/auth setup, resolve credentials from live DB or live auth response only; do not guess from stale artifacts or documentation.
