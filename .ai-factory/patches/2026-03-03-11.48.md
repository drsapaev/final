# Self-profile fallback for backend rate-limit state

**Date:** 2026-03-03 11:48
**Files:** frontend/src/pages/UserProfile.jsx, frontend/src/pages/__tests__/UserProfile.test.jsx
**Severity:** medium

## Problem

`/profile` could fail hard when `GET /api/v1/authentication/profile` returned
`429 Too Many Requests`. In React StrictMode/dev, the page also risked issuing
duplicate initial requests during mount/remount, increasing the chance of hitting
an already polluted backend rate-limit bucket.

## Root Cause

- `UserProfile` fetched self-profile directly on mount with no request dedupe.
- When backend returned `429`, the page treated it as a fatal load error even
  though a cached auth profile was already available in local storage/state.
- In dev, StrictMode remounts can duplicate the first fetch unless the request
  is shared across mounts.

## Solution

- Added module-level single-flight caching for `GET /authentication/profile`
  so concurrent/remounted loads share one request.
- Added a short-lived self-profile cache to avoid immediate duplicate reloads.
- On `429`, the page now falls back to the cached auth profile and shows a
  user-facing warning instead of failing closed.
- Added tests for StrictMode request dedupe and `429` fallback behavior.

## Prevention

- Pages that load user/session data should reuse in-flight requests across
  dev remounts and route transitions.
- Protected UI should prefer degraded rendering from cached auth state over
  blank failure when backend rate limiting is temporarily polluted.
