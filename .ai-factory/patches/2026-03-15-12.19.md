# Backend debug startup diagnostics on Windows

**Date:** 2026-03-15 12:19
**Files:** backend/start_server.py, .vscode/launch.json, .vscode/settings.json
**Severity:** medium

## Problem

Debug запуск backend на Windows падал с `WinError 10013` при старте `backend/start_server.py`.
Дополнительно ручной запуск через `C:\final\.venv\Scripts\python.exe` скрыто ломался на
`ModuleNotFoundError: No module named 'psycopg'`.

## Root Cause

Проблема была составной:
- порт `8000` уже был занят работающим backend, но launcher не сообщал это явно;
- `start_server.py` по умолчанию стартовал с `reload=True`, что под debugger на Windows
  делает запуск менее стабильным и усложняет диагностику;
- workspace setting указывал на Unix-путь `./backend/.venv/bin/python`, поэтому IDE могла
  выбирать неверный интерпретатор вместо `backend/.venv/Scripts/python.exe`;
- отдельный debug-конфиг запускал `uvicorn --reload` в обход launcher и повторял ту же проблему.

## Solution

В `backend/start_server.py` добавлены preflight-проверки и явные `[FIX:START]` сообщения:
- детект локально занятого порта с подсказкой про существующий backend и `BACKEND_PORT`;
- если backend на этом порту уже жив и отвечает health-check, launcher завершает debug-запуск без ошибки;
- автоматическое отключение reload по умолчанию при запуске из debugger;
- понятная ошибка, если активный интерпретатор не содержит `psycopg`.

В `.vscode/settings.json` исправлен default interpreter на Windows-путь к `backend/.venv`.
В `.vscode/launch.json` убран `--reload` из debug-конфига `Debug Backend API`.

## Prevention

- Для Windows debug-стартов не полагаться на `--reload` как дефолт.
- Всегда держать в workspace явный `python.defaultInterpreterPath` для нужного venv.
- Перед запуском dev server проверять, не слушает ли уже тот же порт другой процесс.
- Для launcher-скриптов полезнее короткая диагностическая подсказка, чем сырой traceback.
