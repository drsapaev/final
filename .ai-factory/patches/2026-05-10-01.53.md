# Document backend recovery flags in env example

**Date:** 2026-05-10 01:53
**Files:** backend/.env.example
**Severity:** medium

## Problem

Strict verification found backend feature flags referenced by the recovery code but absent from `backend/.env.example`:

- `ENABLE_FALLBACK_AUTH`
- `ENABLE_SIMPLE_FILE_UPLOAD`

That makes deploy and review checks less reliable because operators cannot see the intended fail-closed defaults from the sample backend environment file.

## Root Cause

The recovery implementation added or hardened fail-closed runtime switches for legacy fallback auth and simple upload behavior, but the backend env example was not updated in the same patch. The flags were documented indirectly in code and some docs, but not in the operator-facing backend env template.

## Solution

Added both flags to `backend/.env.example` with disabled defaults:

- `ENABLE_FALLBACK_AUTH=0`
- `ENABLE_SIMPLE_FILE_UPLOAD=0`

The comments describe their intended break-glass/dev-only nature without adding real credentials or changing runtime behavior.

## Prevention

- When a recovery patch introduces or relies on a settings/env flag, update the closest `.env.example` in the same slice.
- Strict verification should include an env-reference scan against changed files and sample env files.
- Treat fail-closed security flags as operator-facing configuration, even when the default is safe.
