# Sticky EMR write access denial for cardiology save flow

**Date:** 2026-04-09 15:40
**Files:** `frontend/src/hooks/useEMR.js`, `frontend/src/components/emr-v2/EMRContainerV2.jsx`
**Severity:** medium

## Problem

The cardiology EMR flow kept producing `403 Forbidden` on `POST /v2/emr/758`, including autosave retries and manual save clicks. The UI already handled the denial after the fact, but it still kept trying to write until the user navigated away.

## Root Cause

The EMR hook handled `401/403` by returning an `accessDenied` result, but the denial was not persisted as a sticky write-block state. That meant:
- autosave could still arm before the denial was known
- manual save/sign actions could still attempt the same forbidden request again
- the UI did not visibly communicate that the current session could not write to this EMR

## Solution

Added a sticky write-access denial flag inside `useEMR`:
- once any EMR write action receives `401/403`, the hook marks write access as denied for the current visit/session
- subsequent save/sign/amend calls return early without hitting the backend again
- the flag resets when the visit or specialty changes, or when a successful/404 EMR load occurs

The EMR container now:
- disables save/sign actions when write access is denied
- stops autosave entirely in the denied state
- shows a clear no-write-access badge so the UI matches the backend contract

## Prevention

- Treat the first hard access-denied response as a state transition, not just as a one-off error.
- Disable UI actions once the backend contract says the session cannot write.
- Make autosave respect the same denied state as manual save actions.
