# Patch Note - 2026-03-03 10:21

## Context

`POST /api/v1/auth/minimal-login` started returning:

- `429 Too Many Requests`
- detail: `Превышен лимит запросов: 100 за 3600 секунд`

This was incorrect for login. The login limiter is supposed to be `5` requests per `300` seconds, not the generic API limiter `100` per hour.

## Root Cause

`SecurityMiddleware._get_endpoint_type()` recognized:

- `/authentication/login`
- paths ending with `/login`

but did **not** recognize `/api/v1/auth/minimal-login`.

As a result:

- `minimal-login` was classified as generic `api`
- rate limit key became `api:<ip>`
- JSON login flow from Vite also triggered CORS preflight `OPTIONS`
- those preflight requests were counted too

So repeated login attempts eventually hit the generic `100/3600` API bucket instead of the login bucket.

## Changes

### Backend

- Updated `backend/app/middleware/security_middleware.py`
  - classify `/auth/minimal-login` and `/auth/json-login` as `login`
  - skip `OPTIONS` requests from rate-limit accounting
  - add `[FIX:RATE_LIMIT]` log for `minimal-login` classification

### Tests

- Updated `backend/tests/test_security_middleware.py`
  - added regression test for `/api/v1/auth/minimal-login`
  - added regression test to ensure `OPTIONS` preflight does not consume login quota
  - added classifier test for `minimal-login`

## Verification

- `python -m pytest backend/tests/test_security_middleware.py -q`
  - result: `15 passed`

## Expected Outcome

After backend restart:

- `/api/v1/auth/minimal-login` will use the login bucket (`5/300`)
- preflight `OPTIONS` requests will no longer burn rate-limit slots
- the incorrect `100/3600` lockout for login should stop happening
