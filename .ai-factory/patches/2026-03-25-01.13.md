# Notification API contract parity for inbox/realtime minimum

**Date:** 2026-03-25 01:13
**Files:** frontend/src/api/endpoints.js, frontend/src/api/services.js, backend/tests/unit/test_notification_endpoint_inventory.py
**Severity:** medium

## Problem

Frontend notification entry points expected endpoints that are not implemented in the backend `/notifications` router (list by `/notifications`, get by id, mark read, mark all read, notification types). This creates contract drift and future regression risk.

## Root Cause

The frontend `notificationsService` and `API_ENDPOINTS.NOTIFICATIONS` block were modeled as a generic CRUD surface, while the backend notifications domain evolved toward history/send/settings/templates and registrar/fcm-specific flows in separate routers.

## Solution

- Reduced frontend notification constants to the minimum production-ready inbox/realtime set (`history`, `history/stats`, `send`).
- Updated `notificationsService` to use `getHistory` and `sendNotification` as primary supported methods.
- Kept legacy method names as explicit fail-fast stubs with `[FIX:NOTIFICATIONS]` errors, so accidental usage is surfaced immediately instead of silently calling non-existent endpoints.
- Added a static inventory contract unit test that validates the expected minimal backend surface and checks frontend endpoint/service alignment.

## Prevention

- Keep frontend notification contracts as a strict subset of backend-published `/notifications` routes.
- Preserve static inventory tests for contract-sensitive domains to prevent re-introducing dead endpoints.
- Add new frontend methods only after backend routes are published and covered by parity checks.
