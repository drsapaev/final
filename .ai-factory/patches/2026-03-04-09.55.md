# Theme Preference Save Hit Generic API Rate Limit

**Date:** 2026-03-04 09:55
**Files:** backend/app/middleware/security_middleware.py, backend/tests/test_security_middleware.py
**Severity:** medium

## Problem

После исправления клика в меню темы frontend уже локально применял схему, но
`PUT /api/v1/users/me/preferences` падал с `429 Too Many Requests` и сообщением
`Превышен лимит запросов: 100 за 3600 секунд`.

Из-за этого тема не сохранялась в preferences и в консоли появлялся шум от
ошибки сохранения.

## Root Cause

Self-service endpoint `/users/me/preferences` не был включён в session/self
bucket и классифицировался middleware как обычный `api` endpoint. Поэтому он
наследовал общий лимит `100/3600`, который для долгоживущей dev-сессии быстро
загрязнялся фоновой активностью приложения.

## Solution

- Добавлен путь `/users/me/preferences` в session-classification в
  `SecurityMiddleware._get_endpoint_type`
- Добавлены regression tests:
  - GET `/users/me/preferences` использует bucket `600/3600`
  - PUT `/users/me/preferences` использует bucket `600/3600`
  - classifier возвращает `session` для этого пути

## Prevention

- Все self-service endpoints пользователя нужно держать в одном отдельном
  bucket, а не смешивать с общими data/API запросами
- При добавлении нового `/me/*` endpoint сразу покрывать classifier тестом
- Для настроек интерфейса не использовать общий api limit, потому что они часто
  вызываются на boot и при пользовательских переключениях
