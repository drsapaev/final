# CORS disabled by dev startup scripts and incompatible env parsing

**Date:** 2026-03-02 20:20
**Files:** backend/app/core/config.py, backend/app/main.py, backend/start_server.py, backend/run_server_auto.py, backend/run_dev_server.ps1, backend/run_dev_server_auto.ps1, backend/start_server.bat, backend/tests/test_settings.py
**Severity:** high

## Problem

Frontend requests from `http://localhost:5173` to backend endpoints like
`/api/v1/auth/minimal-login` and `/api/v1/password-reset/initiate` failed in
the browser with CORS preflight errors and `net::ERR_FAILED`.

## Root Cause

Two separate issues combined:

1. Dev startup scripts were forcing `CORS_DISABLE=1` before FastAPI settings
   loaded, which disabled `CORSMiddleware` even when `backend/.env` had
   `CORS_DISABLE=0`.
2. Settings expected `BACKEND_CORS_ORIGINS` as a complex list field, but the
   project `.env` and docs used legacy `CORS_ORIGINS` as a comma-separated
   string. `pydantic-settings` tried to parse that value as JSON and failed.

## Solution

Updated all main dev startup paths to default CORS to enabled instead of
disabling it, while still allowing explicit overrides. Added backward-compatible
loading for both `BACKEND_CORS_ORIGINS` and legacy `CORS_ORIGINS`, including
comma-separated parsing. Added `[FIX:CORS]` logs in startup so the effective
mode and allowed origins are visible immediately.

## Prevention

- Do not hardcode security-related env overrides in launcher scripts unless they
  are strictly required for every environment.
- Keep env variable names aligned between code, `.env.example`, and runtime
  documentation.
- Cover env parsing regressions with tests whenever a settings field uses a
  complex type.
