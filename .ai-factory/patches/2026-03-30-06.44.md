# Frontend legacy backend-origin guardrail

**Date:** 2026-03-30 06:44
**Files:** frontend/src/__tests__/noLegacyBackendOrigins.test.js, .ai-factory/RULES.md
**Severity:** medium

## Problem

After fixing one doctor-panel fetch failure caused by a hardcoded `127.0.0.1:8000` URL, we needed a project-wide guardrail so the same regression would not reappear in another frontend module during future work.

## Root Cause

The frontend has a mixed network style: many modules already use shared runtime URL helpers, but older code can still drift back to literal backend origins when copy-pasted. Without an automated guardrail, the regression only shows up later as a browser fetch failure during smoke.

## Solution

Added a runtime-source test that scans `frontend/src` and fails if any non-test file hardcodes the legacy `http://127.0.0.1:8000` or `http://localhost:8000` backend origin. Also updated project rules to require `buildApiUrl` / `buildWsUrl` for browser network calls so origin drift is prevented by convention as well as by test.

## Prevention

- Enforce canonical runtime URL helpers for all browser network calls.
- Keep a legacy-origin guardrail test in the frontend source tree.
- When a fetch failure appears, grep for hardcoded origins before spending time on auth or CORS hypotheses.
