# Network error normalization expanded beyond login

**Date:** 2026-03-30 06:18
**Files:** `frontend/src/utils/networkErrorMessages.js`, `frontend/src/components/auth/loginErrorUtils.js`, `frontend/src/pages/RegistrarPanel.jsx`, `frontend/src/hooks/useDoctorPhrases.js`, `frontend/src/components/chat/LinkPreview.jsx`, `frontend/src/components/emr-v2/sections/ComplaintsSection.jsx`, `frontend/src/utils/__tests__/networkErrorMessages.test.js`
**Severity:** medium

## Problem

Raw browser/network failure text such as `Failed to fetch` was still present in several frontend call sites. In some places it was directly used in conditional logic, and in others it was logged with user-facing wording. That created inconsistent UX and made it easy for the same opaque message to leak back into nearby flows.

## Root Cause

The initial login fix solved only one narrow surface. Other components and hooks still handled network failures locally, each with its own message shape and no shared normalizer. That meant one fix could be undone by another file that still treated fetch failures as raw strings.

## Solution

Introduced a shared `networkErrorMessages` helper that recognizes fetch-style network failures and normalizes validation/detail payloads.

Then applied it to:
- registrar patient lookup from URL
- doctor phrase history suggestions
- link preview fetch logging
- complaints history fetch logging
- login-specific error formatting via the shared helper

Added unit coverage for both the shared helper and the login formatter. Verified that `Failed to fetch` no longer appears in `frontend/src` user-facing code paths.

## Prevention

- Normalize network failures through a shared helper instead of checking raw `err.message` strings in each component.
- Keep UI-facing messages localized and actionable.
- Add regression tests for network failures, validation details, and credential errors.
- Search for raw `Failed to fetch` strings during frontend QA before closing a smoke cycle.
